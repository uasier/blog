Go 命令行工具的基本操作以及帮助：

$ go //与go help 效果相同，获取到使用go CLI tool 的帮助信息

$ go help //获取使用go CLI tool 的帮助信息

$ go help //在go help 后面添加一个具体的命令，可以获得命令对应的介绍与帮助信息

$ go help //在go help 后面添加一个主题，可以获得主题对应的细节信息

通过go help 获取到的相关命令和主题相关注解

7.2 代码风格

“代码必须是本着写给人阅读的原则来编写，只不过顺便给机器执行而已。”这段话来自《计算机程序设计与解释》，很精练地说明了代码风格的作用。

Go语言将代码风格强制统一，一些对于其他语言的编译器完全忽视的问题，在Go编译器前就会被认为是编译错误，比如如果花括号新起了一行摆放，你就会看到一个醒目的编译错误。Go语言的这种做法简化了代码规范问题。

其他的一些编码规范里通常会列出的细节，比如应该用Tab还是用4个空格，这些不在讨论范围之内。

7.2.1 强制性编码规范

  1. 命名

    命名规则涉及变量、常量、全局函数、结构、接口、方法等的命名。Go语言从语法层面进行了以下限定：任何需要对外暴露的名字必须以大写字母开头，不需要对外暴露的则应该以小写字母开头。

    Go语言明确宣告了拥护骆驼命名法而排斥下划线法，在开始Go语言编程时，还是忘记下划线法吧，避免写出不伦不类的名字。

    Ⓜ️ ：以上内容也就是表示你在写Go语言程序时：任何需要对外暴露的名字必须用大驼峰（首字母为大写的驼峰法），不需要对外暴露的则应该用小驼峰（首字母为小写的驼峰法）。

  2. 排列

    Go语言甚至对代码的排列方式也进行了语法级别的检查，约定了代码块中花括号的明确摆放位置。

7.2.2 非强制性编码风格建议

go CLI tool 有代码整理的功能，可以通过这项功能实现代码整洁！

我们并不是非常推荐使用这个工具。毕竟，保持良好的编码风格应该是一开始写代码时就注意到的。第一次就写成符合规范的样子，以后也就不用再考虑如何美化的问题了。

7.4 工程组织

go语言没有固定项目结构

7.4.1 GOPATH

GOPATH这个环境变量是在讨论工程组织之前必须提到的内容。Gotool的大部分功能其实已经不再针对当前目录，而是针对包名，于是如何才能定位到对应的源代码就落到了GOPATH身上。
假设现在本地硬盘上有3个 Go代码工程，分别为 ~/work/go-proj1 、 ~/work2/goproj2和~/work3/work4/go-proj3，那么GOPATH可以设置为如下内容：

export GOPATH=~/work/go-proj1:~/work2/goproj2:~/work3/work4/go-proj3
经过这样的设置后，你可以在任意位置对以上的3个工程进行构建。
